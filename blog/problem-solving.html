Problem Solving

In Sprint 4 - Tech, we learnt about different problem solving techniques, and through the various coding challenges, we had many opportunities to practice some of them!

In this blog post, I will discuss the following:

  • A time when I was 'blocked' by a problem.
  • An example of when I used an elegant solution to a problem
  • A reflection on how confident I feel about using various different problem-solving techniques/processes.

I discuss each of these in more detail below.

A time when I was 'blocked' by a problem.

Part of the Foundations course involves working through a series of problems, which they have called 'kata', that are designed to expose us to different types of coding problems and give us lots of practice in finding solutions to them, using JavaScript.

Although I managed to work my way through them and eventually found solutions to them all, there were many times when I was 'stuck' and had to problem-solve to find a solution.

Many of the kata challenges involved being given a list of data (numbers, words, sentences) and having to somehow run through the list and either make some changes to the list items, or to test the items and create a new list containing only the items that passed the test.

For example:

  • Changing the first letter of all the words in a list to be upper case; or
  • Creating a new list that only included the even numbers from the original list.

The first part of the solution to these types of problems was to figure out how to actually run through a list and look at each item.

I knew how to do it using what can be described as a 'classic' programming technique, where you set up a for 'loop' to run through all the items, but I knew there was a more modern or 'contemporary' technique, which was introduced to JavaScript in 2015, when ES6 was introduced.

ES6 is the most recent major revision to JavaScript and it included a number of new features, or methods, to make it quicker and easier to implement solutions to some of the more common programming tasks.

One of common programming tasks that ES6 has provided solutions for, is to make it easier to run through a list of items, and I wanted to learn about these new techniques, so I had to do some problem solving to do this.

Some of the useful, new techniques, (such as the .map array method), involve using an "arrow" function, which is a 'callback' function. Although the format/syntax is much shorter than the 'classic' looping approach (e.g. it is only one line of code) I struggled to get my head around thr structure of it.

I used google and read some articles, but couldn't make sense of them, so I tried YouTube, and found these two videos which really helped:

  • Inbuilt JavaScript method map() explained!

Finding an elegant solution to a problem.

In one of the other challenges, we had to write a function that takes the content of a string, and returns another string, with the letters all in reverse order.

Again, I knew we could do this with a for loop that started at the end of the string and looked at each item in reverse, taking that item and pushing it into a new array. However, I wanted to do it using some of the ES6 methods.

Once again, I turned to the internet, and this time I hit the google jackpot and found an example that solved this exact problem!

This very elegantly combines three methods (.split, .reverse and .join) on only one line of code: 

                    function reverseString(str) {
                      return str.split("").reverse().join("");
                    }
                    reverseString("hello");

Reflecting on problem-solving techniques/processes.

There are many different techniques that can be used to solve coding problems, including:

  • Pseudocode
  • Trying something
  • Rubber ducky method
  • Reading error messages
  • Console.logging
  • Googling
  • Asking your peers for help
  • Asking coaches for help
  • Improving your process with reflection

We were encouraged to investigate a couple of these. I was already familiar with using Google and I find that pseudo code is really helpful, so I chose two that I wasn't sure about.

Firstly, the 'Rubber ducky method', because the name was so intriguing(!), and then 'Reading Error Messages', as it seems obvious.

The Rubber Ducky Method

As the name suggests, this involves talking to an inanimate object (a rubber ducky) and explaining what each line of code is meant to be doing. The act of taking the time to talk through what your code does, line by line, can help you ‘unblock’ and in doing so, you might spot the issue or find a solution yourself. Describing what it is supposed to do and observing what it actually does can shine a light on the issue. This is a phenomenon I’ve encountered many times, where, while I’m in the process of explaining an issue to someone to see if they have an answer, the answer pops into my head!

Read the Error Message

This seemed too obvious to be a “technique”, so I thought I’d research it and see if I was missing something. Often people don’t read these carefully enough and they may miss valuable clues that could direct you to where the issue is. By understanding the errors you receive and connecting the dots between each error message and the solution to the problem, you’ll gain valuable coding skills.

From my research, I learned that there are some key steps that you can take, such as:

  • Dissect the error and learn to spot the key piece of information (aka ‘clue’) within the error message. It will often point you to a part of the code that your should start looking at.
  • Is the solution in the error? If the problem is due to a syntax error, sometimes the compiler will use a symbol to point to exactly where the issue is.
  • It may help to read it aloud (same philosophy as the rubber ducky method!)
  • Check the syntax (you may need to use google)
  • Do a search for the error message (google again). Someone else may have had the issue and someone may have solved it
  • When trouble shooting, change as little as possible each time, to help isolate the problem.

During Sprint 4, I found that the methods I used most often and most successfully were to write pseudo code, use the console log to see what's happening, and to ask google. I was also much more mindful of reading the error messages and I did find that they often pointed me to the right place.

Luckily, I was able to get by with just these approaches, but if I was still blocked, I would probably have turned to Discord and asked my very helpful peers and/or instructors!
